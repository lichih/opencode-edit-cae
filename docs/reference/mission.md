# Mission: 實作「物理座標錨點編輯 (Coordinate-Anchored Edit)」機制

## 1. 背景與核心問題 (The Context)

AI Agent 在使用目前的 `edit` 工具時，面臨一個結構性的挑戰：**LLM 擅長識別模式 (Pattern)，但極度不擅長精確計數。**

雖然 `expectedOldLineCount` 已實作，但 Agent 仍需經過「讀取 -> 視覺計數 -> 填寫參數」的過程。一旦檔案包含大量不可見字元、連續空行或結尾換行符號，計數極易失準。

### 1.1 核心提議
我們提議將 `edit` 提升為 **「物理導航模式」**：
Agent 應直接使用 `Read` 工具回傳的物理行號（如 38-43 行）作為參數，由工具執行 **「座標對齊 + 內容校驗」** 的雙重鎖定。

---

## 2. 提議功能規格 (Proposed Specs)

### 2.1 新增參數
在 `edit` 工具中引入：
*   **`startLine`**: 目標區塊的起始行號。
*   **`endLine`**: 目標區塊的結束行號。

### 2.2 雙重校驗邏輯 (Double-Lock Logic)
當收到座標參數時，工具必須執行以下程序：
1.  **物理定位**：直接擷取檔案中 `[startLine, endLine]` 區間的原始內容。
2.  **語意校驗**：檢查擷取的內容是否與 Agent 提供的 `oldString` **完全一致 (Bit-by-bit)**。
3.  **執行替換**：僅在兩者完全一致時執行 `newString` 的覆蓋。

---

## 3. 重要指示：可行性評估要求 (Feasibility Audit)

**在正式實作程式碼之前，Opencode 開發 Agent 必須提交一份「技術可行性與邊界案例評估報告」，包含：**

1.  **效能評估**：直接跳轉至行號 (Seek by line) 是否比全檔案掃描 (Pattern Search) 具備顯著的效能優勢？
2.  **安全性分析**：
    *   如何防止「Stale Data (過時讀取)」問題？（例如：當行號正確但內容已變動時，必須強制攔截）。
    *   此機制是否能完全取代 `expectedOldLineCount`？或是兩者並存？
3.  **API 相容性**：如何設計參數，讓不支援座標（或不提供座標）的舊式調用依然能正常運作？
4.  **錯誤回饋設計**：當座標與 Pattern 不符時，如何提供最直觀的診斷資訊（例如：顯示該行號區間的實際內容與預期內容的 diff）？

---

## 4. 預期效益 (Expected Benefits)

1.  **零計數錯誤**：Agent 僅需「抄寫」`Read` 工具顯示的數字，完全避開算術偏差。
2.  **最高信任度**：物理定位 (行號) 與語意定位 (內容) 的重疊，提供了檔案編輯的終極安全性保證。
3.  **簡化 Agent 思考過程**：將「計數邏輯」外包給底層工具，讓 Agent 專注於代碼邏輯的撰寫。
