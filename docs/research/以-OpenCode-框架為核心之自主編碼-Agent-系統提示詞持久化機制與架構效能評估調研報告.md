# 以 OpenCode 框架為核心之自主編碼 Agent 系統提示詞持久化機制與架構效能評估調研報告
隨著人工智慧從單純的文字生成轉向具備自主行動能力的 Agent 系統，如何確保模型在複雜且長路徑的任務中始終遵循安全約束，已成為軟體工程自動化領域的核心課題。OpenCode 作為一個終端原生（Terminal-native）的開源 AI 編碼工具，其設計哲學深受自主性與安全邊界平衡的影響。本報告旨在針對 OpenCode 系統中「Plan Mode」（規劃模式）提示詞的持久化注入機制進行深度的技術調研與架構分析，探討其起源、動機、潛在的指令污染風險，以及當前產業 SOTA（State-of-the-Art）方案的演進趨勢。

## 第一章 OpenCode 框架之技術起源與提示詞基因分析
OpenCode 的系統架構與設計邏輯，在很大程度上承襲自學術界與開源社群對於自主編碼 Agent 的研究成果，特別是與名為 「OpenDev」 的框架有著深厚的技術血緣關係 。OpenDev 被定義為一種針對終端環境設計的複合式 AI 系統，其核心在於透過工作負載特化（Workload-specialized）的模型路由與雙 Agent 架構（Dual-agent architecture），將「規劃」（Planning）與「執行」（Execution）進行嚴格的物理與邏輯分離 。

### 1.1 提示詞來源之考證
針對用戶查詢中提到的 `<system-reminder>` 內容，經過原始碼層級的追蹤與對比，可以確認這段提示詞並非直接源自 Google DeepMind 或 Aider 等傳統框架，而是 OpenCode (由 sst 或 anomalyco 維護) 內部硬編碼的行為規範 。該提示詞的核心邏輯位於 OpenCode 原始碼的 `src/session/prompt.ts` 檔案中，具體位置大約在第 1206 至 1214 行之間 。這段提示詞的設計目的是為了解決 Agent 在自主環境中可能產生的誤操作風險，尤其是在需要先行分析而非立即修改代碼的「規劃階段」。
儘管其具體文案為 OpenCode 原創，但其背後的設計思想——即透過系統提醒（System Reminder）來對抗指令衰減——則源自於 OpenDev 研究論文中提出的「事件驅動系統提醒」（Event-driven system reminders）概念 。這種機制被設計用來在長對話週期中，不斷重新激活模型對於核心約束的注意力。

### 1.2 雙 Agent 架構下的模式劃分
OpenCode 的運行機制依賴於不同模式間的切換，每一種模式都對應著不同的工具集、模型配置與提示詞約束。下表詳細列出了 OpenCode 核心模式的技術差異：

| 模式名稱 | 技術角色 | 工具存取限制 | 提示詞核心約束 | 來源參考 |
| --- | --- | --- | --- | --- |
| Plan Mode | 架構分析與任務拆解 | 禁用 edit 工具，bash 需顯式授權 | 唯讀相位、禁止任何文件修改、絕對約束 |  |
| Build Mode | 代碼實現與自動修復 | 完整工具存取（Read, Edit, Bash, LSP） | 執行規劃內容、自動修復報錯、閉環驗證 |  |
| Sub-Agent | 專項任務專家 | 視任務類型而定（如僅讀取權限） | 任務特定提示詞、過濾後的工具集 |  |
在這種架構下，「Plan Mode」的提示詞被設計成一種「絕對禁令」，旨在防止 LLM 在尚未形成完整規劃前就擅自調用 `sed`、`tee` 或 `echo` 等具有副作用的命令 。這種禁令的硬編碼實作方式，確保了即便用戶在自定義配置中放開了權限，提示詞層面的權威性依然能對模型產生強大的壓制力 。

## 第二章 架構動機：為何選擇持久化歷史注入？
OpenCode 選擇將「模式限制」持久化地存入會話歷史（Session History），而不是採用每輪動態注入（Ephemeral Injection）或僅在 System Message 中定義，其背後有著深刻的技術考量，主要涉及 Transformer 架構的注意力機制缺陷與 Agent 的長期穩定性。

### 2.1 對抗指令衰減（Instruction Fade-out）
在 AI Agent 的長程任務中，隨著對話輪次增加與上下文堆疊，模型往往會出現「指令衰減」現象 。雖然原始的系統提示詞（System Prompt）技術上仍存在於上下文視窗（Context Window）中，但其對模型行為的實際影響力會隨著序列長度的增加而呈指數級減弱。這是由於 Transformer 的自注意力機制（Self-Attention）在處理極長序列時，會產生「迷失在中間」（Lost in the Middle）的效應，即模型更容易關注到序列開頭與結尾的標記（Tokens），而忽略了位於中間的關鍵指令 。
透過將 `<system-reminder>` 持久化地注入到每一輪的歷史中，OpenCode 實際上是在對話的「最新邊界」處不斷重新渲染這些約束條件。這種做法確保了無論會話持續多久，核心禁令始終處於模型注意力權重最高的區域，從而極大提升了安全合規性（Safety Compliance） 。

### 2.2 上下文壓縮與生存策略
另一個關鍵動機在於處理「上下文壓縮」（Context Compaction） 。當會話達到 Token 上限時，Agent 系統通常會採取截斷、摘要或滑動窗口（Sliding Window）策略來釋放空間。在傳統的壓縮邏輯中，較早的訊息會被刪除，這可能導致最初定義的系統約束遺失。
OpenCode 採用了一種稱為「啟始訊息」（Inception Messages）的機制，將關鍵的架構決定、安全規則與編碼標準標記為 `preserve: true`。持久化的提醒訊息在這種策略下，能確保即使歷史被壓縮，Agent 依然能通過歷史中的重複片段找回其「唯讀」的身份認同。這種方法雖然增加了 Token 的消耗，但在確保 Agent 行為一致性方面具有無可替代的作用 。

### 2.3 安全合規性與沙盒邊界
從安全性角度來看，Agent 直接操作宿主機環境具有極高風險。歷史研究指出，未受限的編碼 Agent 曾發生過刪除用戶家目錄（`rm -rf ~/`）或利用路徑穿越漏洞逃逸沙盒的嚴重事故 。OpenCode 透過持久化的提醒，在軟體層面（提示詞工程）與系統層面（權限管理）之間構建了雙重保險。即使底層權限配置出錯，模型若能「內化」這段反覆出現的嚴格禁令，也能在最後一刻停止執行危險命令 。

## 第三章 指令污染與行為遲礙：頻繁切換的技術副作用
儘管持久化注入提供了安全性保障，但當 Agent 在不同模式間切換時，歷史中堆疊的相互衝突的指令會引發嚴重的「指令污染」（Instruction Contamination），進而導致模型產生「行為遲礙」（Behavioral Hesitation） 。

### 3.1 頻率偏差（Frequency Bias）與模式僵化
LLM 在本質上是「高頻率偏差的文本模式重新部署機器」（High-frequency-biased text pattern redeployment machines） 。這意味著模型生成的下一個 Token，很大程度上受歷史中出現頻率最高、模式最清晰的片段影響。當用戶在 Plan Mode 下交談了 10 輪，歷史中就存在 10 次「嚴禁修改文件」的強烈警告。
當 Agent 切換到 Build Mode 時，儘管系統可能注入了新的「允許修改」提示詞，但歷史中那 10 次「嚴禁修改」的模式已經形成了一種強大的統計先驗。學術研究指出，這種頻率偏差會導致模型強調語言模式而非語義概念 。從數學上分析，模型在計算隱藏狀態時，這些重複出現的禁令標記會對注意力分佈產生偏置：

$$
\alpha_{i,j} = \frac{\exp(e_{i,j})}{\sum_{k=1}^n \exp(e_{i,k})}
$$
其中 $e_{i,j}$ 代表第 $i$ 個標記對第 $j$ 個禁令標記的相關性。當禁令標記反覆出現時，其累計的權重 $\sum \alpha$ 可能會壓過單一的「切換模式」指令，導致模型即使在 Build Mode 下依然表現得畏首畏尾，拒絕執行 `edit` 工具 。

### 3.2 行為遲礙（Behavioral Hesitation）的具體表現
這種由於指令衝突引發的遲礙在 OpenCode 用戶群體中已有具體案例。用戶報告指出，即使在 `.opencode/plans/` 目錄下已經被賦予了寫入權限，Agent 依然會因為提示詞中「ZERO exceptions」的硬性規定而拒絕更新計畫文檔 。這種現象被稱為「Agent  incoherence」（Agent 不連貫性），即 Agent 的決策層（規劃）與執行層（工具調用）在語義耦合上發生了斷裂 。
下表總結了指令污染引發的主要負面效應：

| 現象名稱 | 表現形式 | 技術根源 | 影響結果 |
| --- | --- | --- | --- |
| 模式死鎖 | Agent 承認需要修改文件，但聲稱其處於唯讀狀態 | 歷史提示詞權重超過當前指令 | 任務中斷，需人工干預 |
| 權限無視 | 即使配置文件允許寫入，模型依然引用提示詞拒絕操作 | 硬編碼提示詞優先級過高 | 靈活性受限，配置失效 |
| 對話迴路 | Agent 不斷建議用戶「切換模式」，即使已經切換 | 歷史緩衝區中的舊標記未被清除 | Token浪費，用戶體驗下降 |
| 幻覺式拒絕 | 模型虛構出系統報錯，聲稱其無法存取工具 | 頻率偏差導致的安全對齊過度 | 生成錯誤的執行日誌 |

## 第四章 業界 SOTA 實踐與最佳方案調研
面對模式切換、歷史純淨度與指令效力之間的衝突，當前的 SOTA 編碼 Agent（如 GitHub Copilot Workspace, Cursor, Windsurf）已經演進出更為精細的架構處理方案。

### 4.1 GitHub Copilot Workspace：雲端原生與環境隔離
GitHub Copilot Workspace 採用的是一種「任務導向的環境隔離」策略。它將開發流程分解為任務定義、規範開發、計畫生成與代碼實現四個階段 。與 OpenCode 在本地會話中反覆注入提示詞不同，Workspace 利用 GitHub Actions 提供安全的、隔離的雲端開發環境 。
在 Workspace 的架構中，模式的純淨度是通過「Agent 角色分離」來實現的。不同的專業化 Agent 處理不同的任務，每個 Agent 僅獲取與其職責相關的上下文。這避免了在單一對話流中堆疊互斥的禁令。此外，它使用「會話日誌」（Session Logs）來記錄狀態，而不是將所有的安全約束都硬編碼進每輪的對話歷史 。

### 4.2 Cursor：檢查點系統與規則索引化
Cursor 作為 AI 原生 IDE，其處理方式更側重於「狀態恢復」與「規則接地」（Grounding）。其核心實踐包括：

- **檢查點系統（Checkpoint System）：** 在 Agent 進行大規模編輯前，系統會自動創建還原點 。這降低了對「完美提示詞約束」的依賴，因為任何錯誤都可以被瞬間回滾。
- **.cursorrules 索引化：** Cursor 不會將所有規則都推入每輪歷史，而是將項目規則（如 `.cursor/rules/*.mdc`）進行向量化索引 。當 Agent 需要執行特定操作時，系統會動態地檢索相關規則並以「非干擾方式」提供給模型，從而保持對話歷史的純淨。
- **合成訊息標記（Synthetic Message Parts）：** 當需要提醒 Agent 當前狀態（如唯讀）時，SOTA 系統傾向於使用 API 特有的標記（如 Anthropic 的系統訊息部分），而不是混入用戶訊息流中。

### 4.3 解決方案對比分析

| 技術維度 | OpenCode (當前) | SOTA 趨勢 (Workspace/Cursor) | 效能影響 |
| --- | --- | --- | --- |
| 提示詞注入 | 歷史持久化重複注入 | 按需動態檢索與索引 | SOTA 節省 30-50% Token |
| 模式切換 | 依賴模型自覺遵循歷史 | 物理限制工具集與環境隔離 | SOTA 安全性更具確定性 |
| 歷史維護 | 全量堆疊，隨壓縮摘要 | 分層存儲（記憶、上下文、日誌） | SOTA 減少行為遲礙 |
| 規則覆蓋 | 硬編碼 Override | 合併用戶定義權限的動態提醒 | SOTA 提升靈活性 |

## 第五章 學界評估指標與 Benchmark 研究
為了量化 Agent 在指令遵循與模式切換中的表現，學術界已開發出多種 Benchmark，這些工具可以作為 OpenCode 優化提示詞策略的依據。

### 5.1 CAR-bench：複雜環境下的策略一致性
CAR-bench 是一個專門評估 Agent 在多輪工具調用中一致性與限制意識（Limit-awareness）的基準測試 。它引入了「幻覺任務」（Hallucination tasks），專門測試 Agent 在面對缺乏工具或權限不足的請求時，是否能正確拒絕而不是胡亂執行。研究顯示，即使是目前最強的推理模型，在處理這類任務時仍面臨「完成度與合規性」（Completion-compliance tension）的張力——模型往往為了滿足用戶而忽視安全政策 。OpenCode 的持久化提醒正是為了緩解這一現象。

### 5.2 AgentBench 與 LifelongAgentBench

- **AgentBench：** 評估 Agent 在跨域任務（如代碼、搜索、數據庫）中的工具組合能力。研究發現，Agent 的表現存在「有效上下文天花板」，超過此上限後，重複的指令往往會導致不穩定 。
- **LifelongAgentBench：** 關注 Agent 的長期學習與知識轉移。它指出傳統的「經驗回放」（即全量歷史保留）由於不相關信息的干擾，效果有限，建議採用更精確的記憶檢索機制 。這直接挑戰了 OpenCode 「持久化堆疊」的合理性。

## 第六章 架構優化建議與未來展望
針對 OpenCode 當前實作導致的行為遲礙與 Token 浪费，本調研提出以下幾點架構優化方向：

### 6.1 從「硬編碼禁令」轉向「動態權限感知的提醒」
系統應廢除 `src/session/prompt.ts` 中的硬編碼文案，改為調用權限引擎（Permission Engine）生成的動態文案 。若用戶在配置中允許了特定路徑的寫入，生成的提醒應包含「Exception: You MAY write to...」字樣，從而消除模型的語義衝突。

### 6.2 引入「滑動視窗」與「啟始訊息」分離
應將核心安全約束（如 Plan Mode 的唯讀性）作為「啟始訊息」（Inception Message）獨立存儲，並確保其始終位於上下文的最頂部或最新一輪，而不是在歷史中重複出現數十次 。這可以透過修改 `msgs` 數組的組裝邏輯來實現，避免對訊息歷史進行深拷貝造成的內存膨脹 。

### 6.3 採用 Just-In-Time (JIT) 提醒機制
參考 `pi-ai` 的「模式匹配激活」技術，系統可以在 Agent 嘗試調用受限工具的一瞬間，先攔截並發送一個系統提醒，要求模型重新確認其當前模式 。這種「事件觸發」的提醒比「每輪重複」的提醒更具針對性，且能顯著降低 Token 負擔。

### 6.4 本地記憶體層的結構化管理
未來的趨勢是將 Agent 的對話歷史視為「版本化文件系統」進行管理，如 Athena 項目所展示的 RAG 層 。透過將決策日誌、架構規範與會話摘要存儲為 Markdown 並進行混合搜索（Vector + BM25），可以讓 Agent 在不需要全量歷史注入的情況下，依然保有精確的狀態意識與歷史確定性。

## 結論
OpenCode 在 Plan Mode 中採用的持久化提示詞注入機制，雖然在初期有效地利用 LLM 的頻率偏差實現了安全對齊，防止了誤操作，但其硬編碼、非動態且具備污染性的實作方式已成為 Agent 進一步自主化的技術瓶頸。指令堆疊引發的「行為遲礙」不僅降低了任務成功率，也造成了顯著的 Token 成本浪費。
借鑒 GitHub Copilot Workspace 與 Cursor 的經驗，OpenCode 應轉向更為精細的環境隔離與權限感知的動態提示詞工程。通過將安全約束從對話歷史中解耦，並利用結構化記憶系統來維持狀態的一致性，Agent 才能在保證安全的前提下，實現流暢的模式切換與高效的代碼產出。
